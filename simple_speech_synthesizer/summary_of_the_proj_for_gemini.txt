I'm working on a simple vocal synthesis system. The project's goal is to create a very simplified, fully semantically parametric (no AI), fully-synthesized (no cutting together audio files) vocal synthesizer, where I try to simplify human speech to as little amount of parametric dimensions as possible. It's been kinda tough, and there are already quite a few dimensions, but it should be fine hopefully.
The synthesizer itself (not yet coded) is based on the source-filter model of speech. There are 2 sources: a voice source (with additional jitteriness in F0 fundamental frequency; this creates vowels, the voiced component of consonants etc.), and a noise source (for breathiness, the voiceless component of consonants etc.). I recently split the voice source in two: there's an oral tract (aka. passageway), and a nasal tract. These act as if they where sound sources of their own, so they are the ones summed up in actual output: oral cavity + nasal cavity + filtered noise source. The filtering is a bit more complex though.
Essentially, I simplified down tongue position determined formants as "vowel formants" (based on the acoustic analysis of vowels, as done by things like the vowel space). These are, naturally, used in vowels, and for the voiced component in voiced consonants and also in vowels. (I'll get into more nitty-gritty details later.) The formants are stored as simple band-pass filter settings with a central frequency (F1, F2, F3), bandwidths and "importance" for each (importance means semantic amplitude, so not an absolute amplitude, rather more like a percentage, but the actual behaviour is to be determined). I thought I'd also approximate the voiceless component of fricatives in a similar manner, aka. the turbulence-induced hissing sound; I use the same "formant" construct and store these "formants" as constriction_formants. (I know it may be inaccurate to call them formants, but in this project, "formant" is already not quite a formant... it is a (resonant) band-pass filter on the voice/noise source, so it may be fine in the context of how I'm simplifying speech.) There are no specific nasal_formants stored, only one nasal_murmur: 250 (Hz), and the nasality effect is achieved by modifying the existing formants (both vowel_ and constriction_formants). There are two parameters that control the presence of these specific traits for any given phoneme: constriction and nasality. They are multipliers/percentages from 0 to 1. A constriction of 0 is essentially no constriction, aka. a vowel, and a constriction of 1 means full closure of the oral tract, like before releasing a /t/. Somewhere in between live all the turbulent-induced stuff, like /s/, /x/, /ʃ/ etc. Also, stops are just modelled as fricatives with stronger attack-like effect. Nasality is meant to be a simplification of velum closure. A value of 0 means velum is closed (no nasality), a value of 1 means velum is open (nasality). Because this is meant to be used for nasal consonants like /n/ and /m/, as well as nasalization of other consonants and vowels, I had to separate the voice channel to two separate audio channels: oral tract and nasal tract (the two cavities). Though I'm starting to reconsider this... for simplification. The nasality parameter already would attenuate formants in the oral tract to model anti-formant and other attenuating behaviour in a simple way, and the nasal tract output also has to be modified by the oral tract's vowel_formants in order to differentiate different tongue positions with nasal consonants (like /n/ vs /m/ vs /ɲ/, I know the first two are not about tongue position but the simplification should take care of that too). So nasal consonants would be: constriction = 1, nasality = 1... maybe, but I'll have to redo that.
One of the core ideas for this project was to model coarticulation in a very-very simplified manner: by just simulating the parameters moving through some n-dimensional phase-space(s). That parenthesized s means that not every parameter has to be put in one phase space, making a huge phase space. I don't want to necessarily couple together certain parameters. But some may be better coupled, like vowel_formant_frequencies, so they can move through an n-dimensional (for 3 formants, 3 dimensional) space with a maximum velocity towards targets. This is also a core idea: targeting. A real human mouth is 1. not able to teleport from one configuration of the speech apparati to another 2. lazy, not that fast all the time. This means that I could model this behaviour by using target states in the phase space(s) that the parameters want to reach through some simplified physics (right now it's a PD controller). If I set the parameters right, I can get some coarticulation with specifically coding any edge cases in. (In fact, I will not do that.) This may only be good for lower levels of coarticulation though. What I wanted to also do is to model the fact that the tongue not only cannot teleport, but it often slurs together stuff. This may be best done by also having surrounding phonemes modify their respective parameters (formants, any other parameters I mentioned, but I only do formants for now). This is a higher-level, phoneme-level coarticulation.
The system works like this: it is made up of layers of processing, where each layer doesn't know of the previous layer's inner workings and cannot modify it. It is a strict conveyor belt-like thing. These are the layers (many of them are to be developed later, I mostly will do the lower-level ones now):
- NLP layer (this is really just a bonus thingy, this is a vocal synth)
- timing layer (maybe it even has to be merged with the previous layer; this determines all the phonetic timing for the different phonemes, start and end times)
- targeting layer (produces acoustic phase space targets from the timed phonemes; essentially it creates "mouth position goals" (of course, this is all parametric acoustic synthesis, there is no mouth model, there is no 3d audio simulation); this is the layer that also manages higher-level coarticulation, because here we still have knowledge of phonemes)
- acoustic_state layer (a bit of a bad name but I'll leave it; it's the parametric simulation layer, the one that achieves lower-level coarticulation and continuity between phonemes, so it doesn't sound robotically choppy; this layer doesn't know phonemes anymore, only targets)
- realization (merges the higher-level, more semantic parameters down to synthesizer parameters; this needs to happen because the synthesizer layer (see later) cannot understand "breathiness" or "nasality"; those have to be merged as deltavalues into other, more simpler parameters, like formants; I'm leaving out a lot of stuff by the way, I have even more parameters)
- pyo_adapter (converts my proprietary Envelope class to something pyo can handle; this is the synthesis library I use to synthesize the speech from scratch)
- synthesis (the actual synthesis layer; it knows nothing about targets, definately not phonemes; it only knows envelopes to set it's audio components with over time. It's also meant to be very simplistic. Right now, it's literally a Blit source with a weak-falloff-low-pass filter (for spectral tilt) and some Reson (resonant band-pass filters) for formants; there's also white noise with filters similarly... it may get a bit more complex, but it should maybe stay relatively simple; that's why the realization layer exists: so that this synthesis layer can be very simple; it doesn't have to understand "breathiness", "gender" (vocal gender), "nasality" etc.)
All of the phonemes for all languages would be stored with unique phoneme IDs in lookup tables. These contain a phoneme's manner of articulation (which is managed by a function generating sub-targets for special consonants, such as stops or trills getting their special production cycles (closure, buildup, release) or (whatever a trill needs to be produced, but it's cyclic).
